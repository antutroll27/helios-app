## Bug 1 — `gScaleRaw` Typed as `any`

**File:** `src/stores/spaceWeather.ts`
**Line:** 27
**Severity:** Medium — disables type checking on downstream reads of `gScaleRaw`

### Current Code

```ts
const gScaleRaw = ref<Record<string, any>>({})
```

### Root Cause

`GScaleEntry` interface is defined at lines 12–19 of the same file and already used as
the return type annotation in `fetchGScale()` (line 128: `const data: Record<string, GScaleEntry>`).
The `ref` declaration was not updated when the interface was added, so `any` escapes into
the computed `gScale` accessor.

### Fix

Change the type annotation on the `ref` to match the existing interface:

```ts
const gScaleRaw = ref<Record<string, GScaleEntry>>({})
```

### Verification

- `npm run build` must pass with zero TypeScript errors
- `gScaleRaw.value` access inside `gScale` computed (lines 32–36) should now have full
  type inference without any `any` widening

## Bug 2 — `ProtocolCard` Missing `'passed'` Status Label

**File:** `src/components/ProtocolCard.vue`
**Line:** 85
**Severity:** Medium — passed cards display incorrect label `'· UPCOMING'` instead of `'· PASSED'`

### Current Code

```html
{{ isActive ? '● ACTIVE' : '· UPCOMING' }}
```

### Root Cause

The ternary only has two branches. The `isPassed` computed (line 25) is already used to
apply the `.card--passed` CSS class (opacity 0.3), but the status label text was never
updated to handle the third state. Any card with `status === 'passed'` renders the label
`UPCOMING` — visually inconsistent with the dimmed card appearance.

### Fix

Extend the ternary to a three-way expression:

```html
{{ isActive ? '● ACTIVE' : isPassed ? '· PASSED' : '· UPCOMING' }}
```

No script changes required. `isPassed` is already declared at line 25:

```ts
const isPassed = computed(() => props.status === 'passed')
```

### Verification

- A card with `status === 'passed'` must render `· PASSED` in the header badge
- A card with `status === 'active'` must still render `● ACTIVE`
- A card with `status === 'upcoming'` must still render `· UPCOMING`

## Bug 3 — `ProtocolTimeline` Statuses Non-Reactive to Time

**File:** `src/components/ProtocolTimeline.vue`
**Lines:** 21–34
**Severity:** Medium — protocol card statuses are computed once on render and do not
update as time passes; a card transitions from `upcoming` to `active` to `passed` only
when Vue triggers an unrelated re-render

### Current Code

```ts
function getStatus(time: Date, endTime?: Date): 'upcoming' | 'active' | 'passed' {
  const now = Date.now()   // ← not a reactive dependency
  ...
}
```

### Root Cause

`Date.now()` is a plain function call with no reactive dependency. Vue's reactivity
system does not track it; the template re-evaluates `getStatus()` only when a reactive
prop changes (e.g., a new protocol is fetched). `solar.now` is a `ref<Date>` that
is updated every 60 seconds by a `setInterval` inside `useSolarStore`, making it the
correct reactive clock for the whole application.

### Fix

1. Import the solar store at the top of `<script setup>`:

   ```ts
   import { useSolarStore } from '@/stores/solar'
   ```

2. Instantiate it:

   ```ts
   const solar = useSolarStore()
   ```

3. Replace `Date.now()` with `solar.now.getTime()` inside `getStatus`:

   ```ts
   function getStatus(time: Date, endTime?: Date): 'upcoming' | 'active' | 'passed' {
     const now = solar.now.getTime()
     const t = time.getTime()
     if (endTime) {
       const e = endTime.getTime()
       if (now > e) return 'passed'
       if (now >= t) return 'active'
       return 'upcoming'
     }
     if (t < now) return 'passed'
     if ((t - now) / 60_000 <= 60) return 'active'
     return 'upcoming'
   }
   ```

Because `solar.now` is a `ref`, Vue tracks it as a reactive dependency of the template
expression that calls `getStatus(item.time, item.endTime)`. The template re-evaluates
automatically on every 60-second tick.

### Verification

- Manually set system time to within 1 minute of a protocol window start; confirm the
  card transitions to `ACTIVE` on the next 60-second tick without a page refresh
- No new imports beyond the solar store are required
- `npm run build` must pass with zero TypeScript errors

## Bug 4 — `solarPhase` Unreachable `'Astronomical Night'` Branch

**File:** `src/stores/solar.ts`
**Lines:** 81–83
**Severity:** Medium — dead code; `'Astronomical Night'` can never be returned

### Current Code

```ts
if (t < dawnMs || t > nightMs) return 'Night'
if (nightStartMs && t >= nightStartMs && t < dawnMs) return 'Astronomical Night'
return 'Night'
```

### Root Cause

The condition `t < dawnMs || t > nightMs` at line 81 catches every case where `t < dawnMs`.
The `'Astronomical Night'` branch on line 82 requires `t < dawnMs` (its second clause),
but by the time execution reaches it that case has already returned `'Night'`.
The `nightStartMs` value is sourced from `times.value.nightEnd` (line 70) — a SunCalc
field that represents the end of astronomical night / start of astronomical dawn. Its
intent was to carve out an `'Astronomical Night'` sub-phase between astronomical night
start and civil dawn, but the guard order prevents it from ever firing.

### Analysis

Reviewing all callers of `solarPhase` across the codebase: it is consumed in
`GlobeOrbitalContext.vue` and `useCobeGlobeData.ts` for display-only HUD text. The value
`'Astronomical Night'` is never mapped to a distinct visual state — both consumers treat
any unrecognised phase the same as `'Night'`. The branch was likely an early idea that
was never wired to a distinct UI representation.

### Fix (Preferred — Remove Dead Branch)

Remove the unreachable branch and the unused `nightStartMs` variable. The final two
lines collapse into a single fallback:

```ts
if (t < dawnMs || t > nightMs) return 'Night'
return 'Night'
```

Which simplifies to removing the unreachable check entirely. The `nightStartMs` variable
declaration at line 70 can also be removed:

```ts
// Remove this line:
const nightStartMs = times.value.nightEnd ? times.value.nightEnd.getTime() : 0
```

After the fix the tail of `solarPhase` reads:

```ts
if (t > duskMs && t <= nightMs) return 'Nautical Twilight'
if (t < dawnMs || t > nightMs) return 'Night'
return 'Night'
```

The final `return 'Night'` is a safe exhaustive fallback for any remaining edge case
(e.g., polar regions where SunCalc returns `Invalid Date` for some phases).

### Alternative Fix (Retain `'Astronomical Night'` as a Distinct Phase)

If a future UI panel wants to display `'Astronomical Night'` as its own state, the
correct branch ordering is:

```ts
// Astronomical Night must come BEFORE the generic Night guard
if (nightStartMs && t >= nightStartMs && t < dawnMs) return 'Astronomical Night'
if (t < dawnMs || t > nightMs) return 'Night'
return 'Night'
```

This is lower priority. The preferred fix is the simpler removal.

### Verification

- `solarPhase` computed returns one of: `'High Sun'`, `'Solar Noon'`, `'Day'`,
  `'Civil Twilight'`, `'Nautical Twilight'`, `'Night'` — all reachable
- `npm run build` passes with zero TypeScript errors
- No changes to the `solarPhase` return type or exported store interface
