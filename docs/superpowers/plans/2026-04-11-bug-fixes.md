## Task 3: Remove dead `'Astronomical Night'` branch from solar.ts

**Files:**
- Modify: `helios-app/src/stores/solar.ts` lines 70 and 82

**Context:**

Current tail of `solarPhase` computed (lines 70, 80–83):
```ts
const nightStartMs = times.value.nightEnd ? times.value.nightEnd.getTime() : 0
// ...
if (t < dawnMs || t > nightMs) return 'Night'
if (nightStartMs && t >= nightStartMs && t < dawnMs) return 'Astronomical Night'

```

The condition `t < dawnMs || t > nightMs` at line 81 catches every case where `t < dawnMs`. The `'Astronomical Night'` branch on line 82 requires `t < dawnMs` (its second clause), but by the time execution reaches it that case has already returned `'Night'`. The branch is unreachable.

`'Astronomical Night'` is never mapped to a distinct UI state in any consumer (`GlobeOrbitalContext.vue`, `useCobeGlobeData.ts`). The fix removes the dead variable and branch.

- [ ] **Step 2: Remove the unreachable `'Astronomical Night'` branch**

The final `return 'Night'` is a safe exhaustive fallback for edge cases (e.g., polar regions where SunCalc returns `Invalid Date`). Keep it.

## Task 4: Make protocol card statuses reactive to `solar.now` in ProtocolTimeline.vue

**Spec:** `docs/superpowers/specs/2026-04-11-bug-fixes.md` § Bug 3

**Files:**
- Modify: `helios-app/src/components/ProtocolTimeline.vue` lines 21–34

**Context:**

`getStatus` currently calls `Date.now()`:
```ts
function getStatus(time: Date, endTime?: Date): 'upcoming' | 'active' | 'passed' {
  const now = Date.now()
  ...
}
```

`Date.now()` is a plain function call. Vue's reactivity system does not track it. The template re-evaluates `getStatus()` only when a reactive prop changes (e.g., when a new protocol is fetched), not on a time-based tick. Cards therefore only transition state on unrelated re-renders.

`solar.now` is a `ref<Date>` updated every 60 seconds by a `setInterval` inside `useSolarStore`. Using `solar.now.getTime()` instead makes Vue track it as a reactive dependency, and the template re-evaluates automatically on every 60-second tick.

- [ ] **Step 1: Read ProtocolTimeline.vue to find the current import block and `getStatus` function**

Read `helios-app/src/components/ProtocolTimeline.vue` and note the exact `<script setup>` import lines and the `getStatus` function signature and body.

- [ ] **Step 2: Add the solar store import**

In `helios-app/src/components/ProtocolTimeline.vue` `<script setup>`, add after the existing imports:

```ts
import { useSolarStore } from '@/stores/solar'
```

- [ ] **Step 3: Instantiate the solar store**

Add after the existing store instantiations (or after the imports if there are none):

```ts
const solar = useSolarStore()
```

- [ ] **Step 4: Replace `Date.now()` with `solar.now.getTime()` in `getStatus`**

Find the current `getStatus` function and replace `const now = Date.now()` with `const now = solar.now.getTime()`.

The complete updated function should be:

```ts
function getStatus(time: Date, endTime?: Date): 'upcoming' | 'active' | 'passed' {
  const now = solar.now.getTime()
  const t = time.getTime()
  if (endTime) {
    const e = endTime.getTime()
    if (now > e) return 'passed'
    if (now >= t) return 'active'
    return 'upcoming'
  }
  if (t < now) return 'passed'
  if ((t - now) / 60_000 <= 60) return 'active'
  return 'upcoming'
}
```

**Note:** This also upgrades the function to handle `endTime` correctly — when `endTime` is provided, a card is `passed` only when `now > endTime`, and `active` when `now >= time`. This matches the spec and is a required improvement.
